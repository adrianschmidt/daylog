# vitalog — fork rename and automated release pipeline

## Background

The fork has accumulated 80+ commits ahead of `tfolkman/daylog` over the last
two months (sleep commands, structured nutrition DB, CLI food/note/bp
commands, `daylog today`), while upstream has been dormant since 2026-03-28.
The fork's value has shifted decisively toward health tracking (food, BP,
weight, sleep, today summary), and there is no longer a meaningful payoff
to staying upstream-mergeable.

This spec covers a clean break: renaming the project from `daylog` to
**`vitalog`**, and standing up an automated release pipeline using
semantic-release that publishes versioned binaries to GitHub Releases.

## Goals (in scope)

- Rename the project end-to-end: package name, binary, repo URL, config
  directory, database filename, and all user-facing references in code and
  docs.
- One-shot migration command (`vitalog migrate`) that moves an existing
  user's `~/.config/daylog/` and `.daylog.db` to the new locations.
- Automated release pipeline triggered by pushes to `main`, using
  semantic-release for version determination and changelog generation, and
  the existing four-target binary build matrix.
- Conventional-commit discipline going forward; default semantic-release
  rules (feat → minor, fix/perf → patch, BREAKING CHANGE → major).
- Branch-protection-compatible publishing via a dedicated GitHub App
  (`vitalog-release-bot`) added to the Ruleset bypass list.
- Designed so adding crates.io publishing later is a small additive diff,
  not an architectural change.

## Non-goals

- Publishing to crates.io now. Cargo.toml is kept crates.io-friendly so it
  can be enabled later.
- Publishing a Homebrew tap. Possible follow-up.
- Migrating the user's `notes_dir` itself (it's user-configured, not a
  hardcoded path; only the config-dir name and DB filename are renamed).
- Re-recording the demo GIF. The `tapes/` directory is deleted; a new
  recording is a follow-up if desired.
- Changing or rewriting git history.
- Modifying upstream (`tfolkman/daylog`). Upstream remote stays read-only.
- Enforcing conventional commits via a hook (commitlint). Solo project,
  unnecessary friction.

## Decisions

- **Name: `vitalog`.** Real-word adjacent to "vital" + "log"; reflects the
  fork's health-tracking focus. Verified clear on crates.io, Homebrew core,
  and as a meaningful GitHub identifier (similar repos all 0-star, no
  conflicts). Vitalograph is an established UK medical-device brand for
  spirometers; different product class, no expected conflict.
- **Rename scope: full.** Binary, package, repo, config dir
  (`~/.config/vitalog/`), DB filename (`.vitalog.db`). Not the `notes_dir`
  itself, which is user-configured and stays wherever each user has it.
- **Migration: explicit `vitalog migrate` command, idempotent.** Detects
  legacy paths and moves them. Existing installations (only one — Adrian's)
  are auto-detected on every command and a one-line stderr hint points to
  `vitalog migrate`. Until migration is run, the binary falls back to old
  paths so the tool keeps working.
- **Distribution: GitHub Releases only.** Four prebuilt binaries
  (linux x86_64, macOS x86_64, macOS arm64, windows x86_64), no crates.io
  publish in v1, no Homebrew tap.
- **Versioning: semantic-release** with default plugin set. First release
  is **`v1.0.0`** — the rename is itself a breaking change, and 1.0.0
  signals the fork as a standalone project rather than a 0.x of something
  else.
- **Pipeline: gated single workflow, bump-then-build-then-publish**
  (Approach 3-refined Variant A). Cargo.toml and Cargo.lock are bumped
  before the binary build so `CARGO_PKG_VERSION` matches the release
  label; CHANGELOG.md is generated by semantic-release in the publish
  job and committed back to `main`.
- **Atomicity: any job failure → no release.** All jobs check the
  `analyze.outputs.version` guard. The `publish` job is the only place
  that creates a tag or a Release; if anything earlier fails, nothing is
  published.
- **Branch-protection bypass: GitHub App.** Account-attributable PATs are
  rejected to avoid an annual rotation chore and to keep release commits
  authored by a release identity rather than Adrian's user. The App
  `vitalog-release-bot` is installed only on `adrianschmidt/vitalog` and
  is in the Ruleset bypass list with mode "Always".
- **Loop prevention: `[skip ci]` in the release commit message + workflow
  concurrency group.** Belt-and-braces: App-token-pushed commits *do*
  trigger downstream workflows, so `[skip ci]` is load-bearing. The
  `concurrency: { group: release, cancel-in-progress: false }` block on
  the workflow ensures two pushes in quick succession queue rather than
  race.
- **License: keep upstream copyright intact, append vitalog modifications
  copyright.** MIT permits rename and redistribution; we honor the
  original notice and add ours alongside, plus a fork attribution line in
  the README.

## Architecture

### Migration command

`vitalog migrate` — new subcommand defined in `src/cli/migrate_cmd.rs`.
Behavior:

1. **Config directory:** if `~/.config/daylog/` exists and
   `~/.config/vitalog/` does not, rename the directory wholesale.
2. **Database file:** read the (possibly-just-migrated) config to locate
   `notes_dir`. If `{notes_dir}/.daylog.db` exists and
   `{notes_dir}/.vitalog.db` does not, rename it. Sidecars
   `.daylog.db-wal` and `.daylog.db-shm` are renamed alongside if
   present.
3. **Refuses to overwrite:** if both old and new paths exist for either
   resource, exit with an error explaining the situation.
4. **Idempotent:** running twice = no-op with a "nothing to do" message.
5. **Prints summary** of every move, or "nothing to do" if already
   migrated.

**Auto-detection on every other command:** at startup, every subcommand
checks for legacy paths and prints a one-line stderr hint when old paths
exist but new ones don't:

```
Note: Found legacy daylog config at ~/.config/daylog/.
      Run `vitalog migrate` to move it to ~/.config/vitalog/.
```

The command then continues using the *old* paths as a fallback so the
tool stays usable until migration is run. Fallback is bounded: if either
new path exists (config or DB), no fallback occurs for that path —
avoiding "is it reading from old or new?" confusion.

### Release pipeline

Single workflow: `.github/workflows/release.yml` (replacing the existing
tag-triggered workflow entirely).

```
on: push: branches: [main]
concurrency: { group: release, cancel-in-progress: false }

job: analyze   (ubuntu-latest, permissions: contents: read)
  - actions/checkout @ github.sha, fetch-depth: 0
  - actions/create-github-app-token (mint installation token)
  - actions/setup-node
  - npx semantic-release --dry-run, parse "next release version" from log
  - outputs: version (empty string if no release needed)

job: prepare   (ubuntu-latest, needs: analyze, if: version != '')
  - actions/checkout @ github.sha
  - scripts/bump-cargo.sh ${{ needs.analyze.outputs.version }}
  - tar czf bumped-workspace.tar.gz Cargo.toml Cargo.lock
  - actions/upload-artifact: bumped-workspace

job: build     (matrix ×4, needs: prepare, if: version != '')
  matrix:
    - x86_64-unknown-linux-gnu  / ubuntu-latest
    - x86_64-apple-darwin       / macos-latest
    - aarch64-apple-darwin      / macos-latest
    - x86_64-pc-windows-msvc    / windows-latest
  - actions/checkout @ github.sha
  - actions/download-artifact: bumped-workspace, extract over working tree
  - dtolnay/rust-toolchain stable + matrix.target
  - Swatinem/rust-cache
  - cargo build --release --target ${{ matrix.target }}
  - tar czf vitalog-${target}.tar.gz   (or 7z on windows)
  - actions/upload-artifact: vitalog-${target}

job: publish   (ubuntu-latest, needs: build, if: version != '',
                permissions: contents: write, issues: write,
                             pull-requests: write)
  - actions/create-github-app-token (mint installation token)
  - actions/checkout @ github.sha, fetch-depth: 0,
      token: ${{ steps.app-token.outputs.token }}
  - actions/download-artifact: all binary artifacts → ./dist/
  - actions/setup-node
  - npx semantic-release
      env:
        GITHUB_TOKEN: ${{ steps.app-token.outputs.token }}
```

The `publish` job's semantic-release run does the full lifecycle:
re-applies `bump-cargo.sh` via `@semantic-release/exec`, generates
`CHANGELOG.md` via `@semantic-release/changelog`, commits both plus
`Cargo.toml` and `Cargo.lock` via `@semantic-release/git` with the
message `chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}`,
tags the commit `v${nextRelease.version}`, pushes, and creates the
GitHub Release with the binary tarballs/zips from `./dist/` as labeled
assets via `@semantic-release/github`.

Note on the duplicate bump: `prepare` and `publish` both run
`bump-cargo.sh` on the same SHA, deterministically producing identical
output. `prepare`'s artifact is consumed only by the build matrix so
binaries embed the right version; `publish` does its own bump for
semantic-release's `@semantic-release/git` plugin, which expects to
control the commit it makes. The two paths are intentionally
independent rather than `publish` consuming `prepare`'s artifact, to
keep `publish` self-contained and the canonical source of what gets
committed.

All external actions are pinned by commit SHA with an end-of-line `# version`
comment, matching the existing `ci.yml` convention. The specific SHAs are
resolved at implementation time and recorded in the workflow file.

### `release.config.js`

```js
module.exports = {
  branches: ['main'],
  plugins: [
    '@semantic-release/commit-analyzer',
    '@semantic-release/release-notes-generator',
    '@semantic-release/changelog',
    ['@semantic-release/exec', {
      prepareCmd: 'scripts/bump-cargo.sh ${nextRelease.version}'
    }],
    ['@semantic-release/git', {
      assets: ['Cargo.toml', 'Cargo.lock', 'CHANGELOG.md'],
      message: 'chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}'
    }],
    ['@semantic-release/github', {
      assets: [
        { path: 'dist/vitalog-x86_64-unknown-linux-gnu/*', label: 'Linux x86_64' },
        { path: 'dist/vitalog-x86_64-apple-darwin/*',      label: 'macOS x86_64' },
        { path: 'dist/vitalog-aarch64-apple-darwin/*',     label: 'macOS ARM64' },
        { path: 'dist/vitalog-x86_64-pc-windows-msvc/*',   label: 'Windows x86_64' }
      ]
    }]
  ]
};
```

### `scripts/bump-cargo.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail
sed -i.bak "s/^version = \".*\"/version = \"$1\"/" Cargo.toml
rm Cargo.toml.bak
cargo update --workspace --offline 2>/dev/null || cargo update --workspace
```

### GitHub App integration

The App `vitalog-release-bot` (App ID 3607258) is installed on
`adrianschmidt/vitalog` only and is in the Ruleset bypass list. Its
private key and ID are stored as the repo secrets `APP_PRIVATE_KEY` and
`APP_ID`. Workflows mint short-lived installation tokens via
`actions/create-github-app-token`:

```yaml
- uses: actions/create-github-app-token@<sha>  # v1
  id: app-token
  with:
    app-id: ${{ secrets.APP_ID }}
    private-key: ${{ secrets.APP_PRIVATE_KEY }}
```

The token from `${{ steps.app-token.outputs.token }}` is passed to:

- `actions/checkout`'s `token:` input (so subsequent `git push` from the
  runner uses the App identity)
- `npx semantic-release`'s `GITHUB_TOKEN` env var (so its API calls are
  authenticated as the App)

The Installation ID is auto-discovered by the action from the App ID +
the repo context; it does not need to be stored.

### Conventional commits

Standard types accepted: `feat`, `fix`, `perf`, `docs`, `style`,
`refactor`, `test`, `build`, `ci`, `chore`. Bump rules:

- `feat:` → minor
- `fix:`, `perf:` → patch
- `BREAKING CHANGE:` footer (or `feat!:` / `fix!:`) → major
- everything else → no bump

The single historical `polish:` commit is silently ignored
(unrecognized type → no bump). Going forward, those become `refactor` or
`style`. A short "Commit conventions" section is added to
`CONTRIBUTING.md`. No commitlint hook in v1.

### Tag and version baseline

The local `v0.1.0` tag points to upstream's commit and was never pushed
to origin. It is deleted (`git tag -d v0.1.0`) before the first
release. Origin has no tags; semantic-release's first run sees no prior
tags and selects **`1.0.0`** as the initial version.

The first `CHANGELOG.md` will list every conventional commit since the
fork diverged (~80 entries). This is accepted as-is — it serves as a
snapshot of what the fork has added since `daylog v0.1.0`. No hand-seeded
v1.0.0 entry.

## Verification plan

Each phase is verifiable independently before moving on:

1. **Migration command unit tests:** the migration logic is structured so
   the core function takes explicit `from`/`to` paths as parameters; the
   thin shim that resolves real `dirs::config_dir()` paths is glue, not
   logic. Tests use `tempfile`-rooted fake paths to exercise: happy-path
   move, idempotent rerun (nothing to do), refuse-to-overwrite when both
   sides exist, sidecar `.db-wal`/`.db-shm` handled.
2. **Auto-detection unit test:** the detector is similarly path-injectable.
   Tests cover: both old and new absent → no hint; only old present →
   hint printed and old path returned for use; new present → no hint
   regardless of old.
3. **Rename smoke test:** `cargo build` produces `target/release/vitalog`
   binary; `vitalog --version` reports the Cargo.toml version;
   `vitalog readme` prints the renamed README; `vitalog init` in a
   tempdir suggests `~/vitalog-notes/` as default.
4. **Pipeline dry-run on a branch:** add a `workflow_dispatch:` trigger
   to `release.yml` (alongside `push: branches: [main]`) so the pipeline
   can be run manually on a non-main branch. Run it once on the rename
   branch with no conventional-commit changes to confirm `analyze`
   outputs empty version and downstream jobs skip cleanly. Once
   confirmed, the `workflow_dispatch:` trigger can stay (useful for
   future debugging) or be removed.
5. **First real release:** delete local `v0.1.0` tag, merge the rename
   work to `main`, observe semantic-release pick `1.0.0`, build all
   four binaries, commit CHANGELOG.md + bumped Cargo.toml/lock with
   `[skip ci]`, push tag `v1.0.0`, create GitHub Release with four
   labeled binary assets attached.
6. **Branch-protection bypass verification:** confirm the release
   commit was authored by `vitalog-release-bot[bot]`, landed on `main`
   directly without a PR, and that the Ruleset's bypass mode
   "Always" recorded the bypass in the audit log.
7. **Loop prevention check:** confirm that the release commit's
   `[skip ci]` suppresses re-trigger; the workflow runs panel should
   show exactly one `release.yml` run for the originating push, not
   two.
